# Задача 1181 — ExplanatoryNoteEstimate (ПЗ), XSD: тип для `TSignature/Name`

## Контекст
В `ExplanatoryNoteEstimate.xsd` элемент `Name` внутри типа `TSignature` объявлен без явного типа данных.

Пути в документе, где это проявляется:
- `/Construction/Signatures/ComposeFIO/Name`
- `/Construction/Signatures/VerifyFIO/Name`

Замечание: «Объявление пустого элемента, не имеющего явного типа данных… может включать любое содержимое. Необходимо указать тип данных для элемента».

## Как понял задачу (архитектор)
- Сейчас `TSignature/Name` без `type`/`simpleType` фактически имеет тип `xs:anyType`: допускает не только текст, но и произвольные вложенные элементы/атрибуты.
- Это ухудшает качество схемы: валидатор и интеграции не могут гарантировать, что подпись — это строка, и возможны невалидные с точки зрения предметной области структуры, которые формально пройдут XSD.
- Правильная точка исправления — именно `TSignature/Name`, потому что `TExtSignature` расширяет `TSignature`, а значит одна правка закрывает оба пути (ComposeFIO/VerifyFIO).
- Тип разумно сделать строковым. Для подписи/ФИО практичнее `non-empty-string` (в схеме уже есть такой тип), чтобы запретить пустые значения.
