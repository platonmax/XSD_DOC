# Политика именования XSD (Минстрой + общая для проекта)

## Общие принципы
- Читаемость: имена полные и понятные, без непояснённых сокращений (`objectNumber`, а не `num`; `sourceDocumentInfo`, а не `info`).
- Элементы и атрибуты — `lowerCamelCase`.
- Типы (complex/simple) — `lowerCamelCase` + суффикс `Type` / `CodeType` / `IdType`.
- Пространства имён и префиксы — стабильные, не меняем без необходимости.

## Коллекции
- Элемент-коллекция с суффиксом `List` (требование Минстроя): `entityList`, `sectionList`, `itemList`, `volumeList`, `documentationList`.
- Элементы внутри — в единственном числе: `item`, `section`, `documentReference`, `alteration`.

## Идентификаторы и ссылки
- Имена: `…Id` для идентификаторов, `…Ref` для ссылок (элементы/атрибуты в `lowerCamelCase`).
- Типы: `…IdType`, `…RefType`.
- Коды/классификаторы: элементы `…Code`, типы `…CodeType`.

## Основные сущности (шаблон)
- Корень/документ: осмысленное имя (`quantityTakeoff`, `projectDocument`, `request`), тип `…DocumentType` или `…Type`.
- Реквизиты: `documentMeta`, `accessLevelCode`, `objectName`, `objectNumber`, `documentDate`, `customerName`, `signatures`, списки `alterationList`, `estimateList`, `fileList`, `exportDateTime` (адаптируйте под домен).
- Списки бизнес-сущностей: `sectionList/section`, `itemList/item` и т.п.
- Объёмы/значения: `value + unitCode` или `amount + currencyCode`; изменения — `increase/decrease/reason`.
- Документы/ссылки: `documentationList/documentReference[@alterationRef]/fileInternalId|pageNumbers|ifcGuids|property*` (пример структуры, имена адаптируйте, но сохраняйте явность).

## Когда выносить именованный complexType
- Структура повторяется или может использоваться повторно.
- Структура сложная (атрибуты, ключи/ID/IDREF, вложенность) или это отдельное доменное понятие.
- Инлайн допустим только для одноразовых простых контейнеров (1–3 поля, без атрибутов/ID). При сомнениях — выносить в именованный тип.

## Элементы vs атрибуты
- Элементы — основной способ передачи бизнес-данных, вложенных структур, списков, текстов; использовать по умолчанию.
- Атрибуты — для метаданных о самом элементе: идентификаторы/ссылки (`…Id`/`…Ref`), коды версий, MIME-типы, роли; объём атрибутов — небольшой, без многострочных текстов.
- Никогда не помещать крупные тексты и списки в атрибуты; всё содержательное — в элементы.
- Если у сущности есть значение и параметры, значение делаем элементом, параметры — атрибутами (пример: `<file id="..." guid="..."><name>...</name></file>`).
- При сомнениях: использовать элемент, атрибут — только для компактных служебных данных.

## Согласованность между схемами
- Одинаковый стиль имен: суффиксы `Type`, `List`, `Id`, `Ref`, `Code` и `lowerCamelCase` везде.
- Стабильные префиксы/неймспейсы для базовых модулей (например, `bt`, `ct`, `bd`, `cqt`) и предсказуемые пути импортов (`types/*.xsd`).
- Имя корневого XSD совпадает с именем каталога/проекта.

## Документация и ясность
- Раскрывать смысл в имени: `customerName`, `requestDate`, `contractNumber`, `documentVersion` вместо обезличенных `name`, `date`, `version`.
- Для перечислений/классификаторов — держать `Code`/`CodeType`; источник классификатора описывать в документации, а не в имени.
- Для дат — уточнять контекст: `documentDate`, `exportDateTime`, `createdDate`, `updatedDate`.
